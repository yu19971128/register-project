# 模块：挂号订单管理

## 职责
- 提供挂号订单的**查询**、**退号**、**改号**、**状态流转**能力，覆盖管理端（PC）与移动端（H5）两类使用者。
- **管理端（PC）**：
  - 管理员可按「日期 / 科室 / 医生 / 状态 / 患者关键词」多维筛选查看当天及历史挂号订单，支持分页。
  - 可对单个订单执行操作：查看详情、代为退号、修改订单备注。
  - 提供简要统计（当天订单总数、已完成数、已退号数、待就诊数）。
- **移动端（H5）**：
  - 患者可查看本手机号关联就诊人的全部挂号记录，按时间倒序展示。
  - 对「待就诊」状态的订单，可自助申请退号；对「已退号」「已完成」订单仅支持查看。
- **状态机**：`待就诊 (PENDING)` → `已完成 (COMPLETED)` / `已退号 (CANCELLED)` / `已改期 (RESCHEDULED)`，状态流转需保证事务一致性。
- **退号 / 改号核心约束**：
  - 退号：就诊时间 30 分钟前方可退号；退号成功后必须**原子性回滚号源余量**（调用「号源管理」模块的扣减回滚接口）。
  - 改号：等价于「退旧号 → 挂新号」，新号挂号成功后旧号才正式释放；若新号挂号失败，旧号保持原状态。
  - 所有状态变更落库时必须在单一数据库事务中完成。

## 边界
- **不做挂号创建**：新订单的创建属于「当天挂号」模块，本模块只管后续生命周期。
- **不做号源配置/扣减逻辑本体**：号源余量的原子扣减、回滚由「号源管理」模块负责，本模块只调用其接口。
- **不做就诊人档案管理**：就诊人信息的 CRUD 属于「就诊人管理」模块，本模块只读就诊人基础信息用于展示。
- **不做支付 / 退款**：系统不含在线支付，因此无资金流；退号仅释放号源、不涉及金额。
- **不做到院核验 / 叫号**：实际就诊流程（叫号、检查、开药）不在系统范围内，本模块只根据管理员/定时任务将已过就诊时间的「待就诊」订单推进到「已完成」。
- **不做短信 / 微信通知**：退号/改号通知超出当前范围，如需再由独立通知模块处理。

## 依赖模块
- **当天挂号**：复用其创建的 `orders` 表结构，订单主数据由其写入。
- **号源管理**：调用「扣减号源余量」与「回滚号源余量」接口，实现挂号 / 退号 / 改号的号源联动。
- **就诊人管理**：查询订单详情与列表时读取就诊人的脱敏信息（姓名、手机号、身份证）。

## 对外输出
- **REST API**：订单列表查询、订单详情、退号、改号、订单备注更新、统计汇总（详见 `docs/06-api/order.md`，本模块产出后冻结）。
- **内部事件**（可选）：`OrderCancelled`、`OrderRescheduled`，供后续通知模块订阅；当前阶段不强制实现。

## 非功能性要求
- 列表查询 P95 响应时间 < 500 ms（10 万订单量级下，依赖复合索引 `(visit_date, status)`）。
- 退号 / 改号必须在一个数据库事务内完成「订单状态更新 + 号源余量回滚」，禁止两步提交。
- 所有订单变更写入 `audit_log` 审计表（字段设计见数据库设计子步骤 D），保留至少 1 年。
